# Agno 框架深度解析

> 原名 Phidata，2.0 后更名为 Agno。41K+ GitHub Stars，Python，Apache 2.0。

## 一、整体定位

Agno 的标语：**"Build, run, and manage your own agent platform."**

它是目前唯一同时覆盖三层的 Agent 平台堆栈：

```
┌──────────────────────────────────────────────────┐
│              Agno SDK (开发层)                    │
│  Agent · Team · Workflow · Knowledge · Memory     │
│  Tools · Models · Skills · Reasoning · Guardrails  │
├──────────────────────────────────────────────────┤
│          AgentOS Runtime (运行时层)                │
│  Storage · Monitoring · Scheduling · API Server   │
│  Auth/RBAC · Human-Approval · Interfaces          │
├──────────────────────────────────────────────────┤
│      AgentOS Control Plane (管理层)               │
│  Web UI · Traces · Sessions · Teams · Evals       │
└──────────────────────────────────────────────────┘
```

## 二、核心构成详解

### 2.1 Agent — 最小执行单元

**定位**: 一切的基础。一个 Agent 封装了模型、工具、记忆、知识、护栏，是一个自包含的"AI 工作者"。

关键配置参数（100+ 项，按类别）：

| 类别 | 参数 | 作用 |
|------|------|------|
| 模型 | `model`, `fallback_models` | 指定 LLM 及故障切换策略 |
| 工具 | `tools`, `tool_choice`, `tool_call_limit` | 注册 Agent 可调用的函数 |
| 指令 | `description`, `instructions`, `system_message` | 定义 Agent 的角色和行为 |
| 输出 | `output_schema`（Pydantic model） | 强制结构化输出 |
| 记忆 | `memory_manager`, `update_memory_on_run` | 跨会话持久化用户信息 |
| 知识 | `knowledge`, `search_knowledge` | RAG 知识库接入 |
| 推理 | `reasoning`, `reasoning_model` | Chain-of-Thought 推理 |
| 钩子 | `pre_hooks`, `post_hooks` | 运行前后注入逻辑 |
| 护栏 | `guardrails` | 输入输出校验 |

### 2.2 Team — 多 Agent 协作

**定位**: 多个 Agent 组成团队，支持不同的协作模式。

四种 TeamMode：

| 模式 | 工作方式 | 适合场景 |
|------|---------|---------|
| **Route** | 路由模式 — 根据任务内容分发到最合适的 Agent | 不同领域专家各司其职 |
| **Broadcast** | 广播模式 — 所有 Agent 收到同一任务 | 多角度分析同一问题 |
| **Coordinate** | 协调模式 — Agent 间可互相通信协作 | 需要分工合作的复杂任务 |
| **Supervisor** | 监督模式 — 一个 Supervisor Agent 分配任务给子 Agent | 有管理层次的任务流水线 |

### 2.3 Workflow — 工作流编排

**定位**: 比 Team 更精细的控制流，适合有明确步骤顺序的场景。

- 定义 Step（步骤），每个 step 可以调用 Agent 或执行自定义逻辑
- 步骤间共享上下文（RunContext）
- 支持异步/并行步骤
- 运行时可注入人类审批（Human-in-the-loop）

### 2.4 Models — 模型抽象层

**定位**: 统一 40+ 模型提供商的接口，让 Agent 代码与具体模型解耦。

支持厂商覆盖：OpenAI、Anthropic、DeepSeek、Google、Mistral、Groq、Cohere、AWS Bedrock、Azure、Perplexity、Together、Fireworks、OpenRouter、vLLM、Ollama、llama.cpp、xAI(Grok) 等。

国内厂商：DeepSeek、Moonshot(月之暗面)、Minimax、SiliconFlow(硅基流动)、DashScope(阿里通义)、InternLM(商汤)

关键能力：
- **fallback 机制**: 主模型失败自动切换备用模型
- **streaming**: SSE + WebSocket 多模式响应
- **结构化/非结构化输出双通道**

### 2.5 Tools — 工具系统

**定位**: 通过 Toolkit 抽象，让 Agent 能调用外部能力。

Toolkit 结构：

```
Toolkit
 ├── name: str                                   # 工具包名称
 ├── functions: Dict[str, Function]              # 同步函数注册表
 ├── async_functions: Dict[str, Function]        # 异步函数注册表
 ├── instructions: str                           # 工具使用说明（注入 system prompt）
 ├── requires_confirmation_tools                 # 需人类确认的工具有哪些
 ├── stop_after_tool_call_tools                  # 调用后停止 Agent 循环
 ├── show_result_tools                           # 结果需要展示给用户
 └── cache_results: bool                         # 是否启用结果缓存
```

100+ 预置 Toolkit：搜索、文件系统、数据库、Slack、Gmail、Google Drive、GitHub、Jira、Notion、MCP 协议集成等。

### 2.6 Knowledge — 知识库（RAG 全链路）

**定位**: 为 Agent 提供外部知识检索能力，支撑 RAG 场景。

完整链路：

```
源文档（本地/URL/云端）
  → Reader（PDF / CSV / JSON / Markdown / 网页...）
    → Chunking（分块策略：固定大小 / 语义分块 / 递归分块）
      → Embedding（向量化）
        → VectorDB（PgVector / Chroma / Qdrant / Milvus / Pinecone...）
          → Reranker（重排序，提升检索质量）
            → Agent 上下文
```

两种使用方式：
1. **被动 RAG**: `add_knowledge_to_context=True` — 自动将相关知识注入 prompt
2. **主动 Agentic RAG**: `search_knowledge=True` — Agent 自主调用搜索工具检索知识

### 2.7 Memory — 记忆系统

**定位**: 让 Agent 拥有跨会话的"记住用户"能力。

三层记忆架构：

```
Session Memory（会话级）
  └── AgentSession（当前会话消息 + 会话状态）
      └── SessionSummary（会话摘要，跨会话快速回顾）

User Memory（用户级，跨会话持久）
  └── MemoryManager
      └── DB Schema: UserMemory
          ├── memory（记忆内容文本）
          ├── metadata（元数据，如时间戳、来源）
          └── user_id（用户标识）

Learning（自主学习）
  └── LearningMachine
      ├── stores（RAGStore / VectorStore...）
      └── strategies（不同学习策略）
```

关键行为：
- `update_memory_on_run=True`: 每次运行结束时自动从对话提取用户记忆
- `enable_agentic_memory=True`: Agent 可以动态读写用户记忆（作为工具）
- `add_memories_to_context`: 将相关记忆注入上下文

### 2.8 Skills — 技能系统

**定位**: 结构化的技能包，包含指令、参考文档和脚本。

与 [[concepts/Agent Skill|Agent Skill]] 概念类似，都是让 Agent 拥有可加载的外部能力。Agno 的 Skill 偏向指令注入+参考文档模式。

### 2.9 Storage — 持久化存储

**定位**: Agent 运行状态的持久化后端。

支持类型：PostgreSQL、SQLite、MongoDB、Redis、MySQL、DynamoDB、ClickHouse、Firestore、SingleStore、SurrealDB...

存储内容：会话、运行记录、消息、记忆、知识、追踪数据...

### 2.10 AgentOS Runtime — 运行时平台层

**定位**: 从 SDK 到可部署产品的桥梁。

| 组件 | 作用 |
|------|------|
| API Server | 50+ REST endpoints + SSE + WebSocket |
| Scheduling | Cron 定时任务 + 后台作业 |
| Auth/RBAC | JWT 认证 + 多租户隔离 |
| Human Approval | 运行中暂停等待人工确认 |
| Observability | OpenTelemetry 追踪 + 审计日志 |
| Interfaces | AG-UI（聊天界面）/ Slack / Telegram / WhatsApp / Discord / A2A |

## 三、运行时流转——一次 Agent 调用的完整生命周期

### 场景假设

> **场景**: 搭建了一个"碳足迹分析师"Agent，配置：
> - 模型: DeepSeek
> - 工具: 搜索 + 碳因子数据库查询
> - 知识库: 产品 LCA 标准文档（VectorDB）
> - 记忆: 跨会话记住用户偏好（"该用户偏好中国本地碳因子数据"）
> - 输出: 结构化报告（Pydantic Schema）

### 完整执行流程

```
用户输入: "分析这款锂电池的碳足迹，重点关注原材料阶段"
          │
          ▼
┌─ Step 1: Session 初始化 ──────────────────────────────────┐
│  → 创建或恢复 AgentSession                                │
│  → 加载 session_state（用户偏好、历史上下文）              │
│  → 从 DB 加载 session_summary（跨会话摘要）               │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 2: Context 组装 ────────────────────────────────────┐
│  → 系统消息: description + instructions 构成角色定义       │
│  → 记忆注入: 从 UserMemory 加载相关记忆                    │
│    （如 "该用户偏好使用中国本地碳因子数据"）                │
│  → 知识检索: 从 VectorDB 检索 LCA 标准片段                │
│  → 工具声明: search + query_carbon_factor 注册到 model    │
│  → 历史消息注入（若 add_history_to_context=True）          │
│  → session_state 注入（若 add_session_state_to_context）   │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 3: 背景任务启动（异步）──────────────────────────────┐
│  → memory_future: 准备从这次对话提取新记忆                 │
│  → learning_future: 准备学习新模式（若 learning 启用）      │
│  → cultural_knowledge_future: 文化知识提取（若启用）       │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 4: Pre-hooks ───────────────────────────────────────┐
│  → 输入校验 guardrails                                    │
│  → 自定义前置逻辑（权限检查、输入预处理等）                │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 5: Reasoning（可选）─────────────────────────────────┐
│  → Agent 先进行 Chain-of-Thought 推理                     │
│  → 可用独立的 reasoning_model（如更贵的模型专门做推理）    │
│  → 推理步骤写入 messages → 影响后续响应                    │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 6: Model 调用（核心循环，可能多次迭代）──────────────┐
│                                                            │
│    ┌──────────────────┐                                    │
│    │   call_model      │ ◄── fallback_config 检查          │
│    │   (带 tools)      │     主模型失败 → 自动切换         │
│    └────────┬─────────┘                                    │
│             │                                              │
│             ▼                                              │
│    ┌──────────────────┐                                    │
│    │  ModelResponse    │                                   │
│    ├──────────────────┤                                    │
│    │ 选项 A: 文本回应  │──→ 输出完成，跳出循环              │
│    │ 选项 B: 工具调用  │──→ 进入工具执行                    │
│    └──────────────────┘                                    │
│             │                                              │
│       若 B: │                                              │
│             ▼                                              │
│    ┌──────────────────┐                                    │
│    │ 执行工具调用      │                                    │
│    │ search("锂电池")  │                                    │
│    │ query_factor(...) │                                    │
│    ├──────────────────┤                                    │
│    │ 工具结果注入 messages                                  │
│    │ 压缩工具结果  (compress_tool_results)                  │
│    └────────┬─────────┘                                    │
│             │                                              │
│             └────→ 回到 call_model（循环迭代）              │
│                                                            │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 7: 输出后处理 ──────────────────────────────────────┐
│  → output_schema（Pydantic）：将原始输出转为结构化 JSON    │
│  → parser_model：用独立模型解析和结构化输出               │
│  → followups：生成后续建议（如 "想分析使用阶段吗？"）      │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 8: Post-hooks ──────────────────────────────────────┐
│  → 输出校验 guardrails                                    │
│  → 自定义后置逻辑（写入审计日志、发送通知等）              │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 9: 等待后台任务 ─────────────────────────────────────┐
│  → await memory_future：提取 & 存储用户记忆               │
│  → await learning_future：提取 & 存储学习模式             │
│  → await cultural_knowledge_future                        │
│  → 合并后台 metrics 到主 metrics                          │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
┌─ Step 10: Session 持久化 ──────────────────────────────────┐
│  → 更新 AgentSession：消息 + metrics + session_state       │
│  → 创建/更新 SessionSummary（若 enable_session_summaries）│
│  → 关闭所有 tool connections                               │
│  → 返回 RunOutput 给调用方                                │
└─────────────────────────────┬─────────────────────────────┘
                              │
                              ▼
                    用户收到结构化响应 + metrics
```

### Team 场景下的额外流转

以 Route 模式为例：

```
Team.run("分析锂电池碳足迹")
  │
  ├── TeamMode.Route: 路由决策 → "这是产品生命周期分析任务"
  │     └── 路由到 LCA Analyst Agent
  ├── LCA Analyst Agent 执行上述 Agent 完整流程
  │     ├── 调用 search 搜索锂电池原材料数据
  │     ├── 调用 query_carbon_factor 查询碳因子
  │     └── 返回分析结果给 Team
  └── Team 汇总 → 返回给用户
```

Coordinate 模式下 Agent 间可通信：
```
Agent A: "原材料数据不够，需要更多背景"
Agent B: "我查到了供应链数据，传递给你"
Agent A: "收到，整合到计算中"
```

### Workflow 场景下的额外流转

```
Workflow("生成碳足迹报告")
  ├── Step 1: 数据采集 Agent → 搜集原材料数据
  ├── Step 2: 计算 Agent → 调用 LCA 算法计算碳足迹
  ├── Step 3: 人类审批 → 暂停，等待用户确认计算结果
  ├── Step 4: 报告生成 Agent → 生成最终报告 PDF
  └── 完成，返回 WorkflowRunOutput
```

## 四、关键评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 上手难度 | ★★★☆☆ | 声明式配置多，但 API 设计整洁。比 LangGraph 简单，比 Hermes/DeerFlow 复杂 |
| 生产就绪 | ★★★★★ | AgentOS 提供完整 API + UI + 多租户 + 监控 + 审计 |
| 国内模型覆盖 | ★★★★☆ | DeepSeek、Moonshot、Minimax、SiliconFlow、DashScope、InternLM 都有 |
| 灵活性 | ★★★★☆ | 100+ 配置项覆盖大部分场景，但重配置不重编程 |
| 性能 | ★★★★☆ | 内置 checkpointing、结果缓存、工具结果压缩 |
