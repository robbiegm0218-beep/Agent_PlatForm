## 1. 目标与边界

目标是搭建一个**个人使用的 Agent 平台**：用户登录后可像使用 Codex 一样创建、续聊和查看历史对话；可在前台查看并启用代码维护的技能；可查看已接入的应用；可修改个人信息和退出。默认模型为 DeepSeek。

本计划以“真实可用”而不是“界面演示”为验收目标。每一轮对话、刷新页面、重新登录和打开历史会话都必须可用。

本期明确不做：多人/组织、支付、用户上传插件、文件工作区、代码执行沙箱、多 Agent 自动编排。它们保留为后续能力，不进入当前关键路径。

## 2. 当前状态与首要问题

已有原生前端、Python HTTP 服务、SQLite、登录、技能开关、DeepSeek 调用和历史记录骨架。

当前 P0 缺陷是 SSE 对话接口在发送 `done` 事件后仍声明 `keep-alive`，浏览器读取流不会结束，前端 `streaming` 状态也不会复位。因此首轮结果可见，但第二轮发送、历史会话切换和错误恢复会被卡住。

解决原则：本期上游 DeepSeek 使用非流式请求，平台向浏览器分段推送；平台完成一次回答后必须关闭 SSE 响应。前端必须使用 `try / catch / finally` 恢复发送状态，并在失败时保留用户消息和可重试入口。

## 3. 目标架构

```text
浏览器（Codex 风格 UI）
        |
        v
API / 认证层
        |
        +-- 会话与消息服务 ---- SQLite（当前）/ PostgreSQL（后续）
        |
        +-- Agent Runtime
        |     +-- 运行记录与事件
        |     +-- 上下文装配
        |     +-- 技能选择
        |
        +-- Model Adapter ---- DeepSeek Chat Completions
        |
        +-- Skill Registry ---- 代码内置技能（当前）/ MCP（后续）
```

当前阶段保留轻量实现，但代码职责必须按上图分开。不要把数据库操作、Prompt 拼接、模型 HTTP 调用和 SSE 输出长期堆在同一个处理函数中。

## 4. 实施顺序

### P0：修复对话闭环

目的：确保同一对话连续发送、历史对话恢复、刷新后继续对话均稳定可用。

实施项：

- [x] 后端：在 `done` 或 `error` 事件后明确关闭 SSE 响应；移除不必要的 `keep-alive`。
- [x] 前端：`sendMessage` 改为 `try / catch / finally`，无论网络、模型或 SSE 解析是否失败都复位发送按钮和 `streaming`。
- [x] 前端：收到 `error` 后不把空白 assistant 消息写入本地状态；显示“重试”操作。
- [x] 后端：以稳定顺序读取消息，使用 `(created_at, id)` 排序，避免同一秒内多条消息顺序不确定。
- [x] 后端：确认线程归属后才删除线程及其消息，避免越权删除风险。
- [x] 验证：自动化覆盖连续两轮、历史顺序、失败重试；浏览器覆盖真实 DeepSeek 连续两轮与发送状态恢复。

完成标准：以上五项均通过，`runs` 中每次运行最终为 `completed` 或 `failed`，没有永久 `running` 记录。

### P1：会话体验与运行可观测性

目的：让个人日常使用时能判断“正在做什么、失败在哪里、历史是否可靠”。

实施项：

- [x] 建立会话、运行、技能与模型调用的逻辑边界；HTTP Handler 保持为 API 编排入口，后续可直接迁移到独立模块。
- [x] 为运行增加阶段事件：`started`、`model_request`、`completed`、`failed`。
- [x] 新增运行详情抽屉：展示模型、耗时、状态、事件链和错误摘要；不显示 API Key 或完整敏感 Prompt。
- [x] 支持重命名、删除和搜索历史对话；删除前二次确认。
- [x] 自动生成标题：首条消息按规则生成，用户可修改。
- [x] 限制上下文窗口：早期消息压缩为摘要，仅携带最近 12 条，控制 DeepSeek 成本。
- [x] 验证：连续对话、失败重试、运行记录、重启服务与历史持久化已通过自动化回归；长对话按摘要与最近 12 条策略受控。

完成标准：用户不需要打开终端即可判断一次对话是否成功、为什么失败，以及是否能够安全重试。

### P2：技能与应用成为真正的运行能力

目的：前台开关不只是展示，而是可以被审计地影响 Agent 的行为。

实施项：

- [x] 将内置技能迁移到 `server/skills/*.json`，保留代码审核与版本控制。
- [x] 定义技能标准字段：`id`、名称、说明、版本、系统提示、输入限制、是否默认启用、状态。
- [x] 对每一次运行持久化“已启用技能快照”，确保历史回放不受后续技能修改影响。
- [x] 支持会话默认技能与消息级 `@技能` 选择；消息级选择优先，用户关闭技能为不可绕过的硬约束。
- [x] 每次消息仅将当次允许的技能传给模型；询问平台技能时由平台根据有效快照返回，不依赖模型自行猜测。
- [x] 应用页展示模型连接状态、技能库版本和未来工具连接状态；未接入的能力明确显示为“未配置”。
- [x] 验证：端到端回归已执行；会话级技能选择会写入下一次运行快照。

完成标准：所有前台可见的技能都能影响一次具体运行，并且运行后可追溯。

### P3：本地可靠性与安全基线

目的：将“本机能跑”提升为“个人长期稳定使用”，不依赖云数据库、Redis 或外部部署平台。

实施项：

- [x] 使用 PBKDF2-SHA256（标准库、随机盐、高迭代次数）替换 SHA-256；添加会话过期时间与退出所有设备功能。
- [x] `.env` 仅用于本机且保持 Git 忽略；启动状态只显示是否已配置，不输出 Key。
- [x] 保持 HTTPS 证书校验；支持 `certifi` 或指定 CA 文件解决企业代理证书问题。
- [x] 增加请求大小、单次回答长度、超时和本机内存限流。
- [x] 增加结构化本地日志与错误码；日志中不记录 token、邮箱与模型响应内容。
- [x] 增加本地备份与恢复命令：SQLite 备份到项目外指定目录，并可恢复到独立文件验证。
- [x] 验证：安全哈希、健康检查、服务重启和数据库备份恢复已通过；模型网络异常沿用现有错误事件与可重试提示。

完成标准：密钥不进入代码库；常见失败可恢复且对用户有明确提示。

### P4：本地扩展接口

触发条件：需要本地文件处理、受控工具调用或复杂工作流时再启动。个人纯对话使用不需要提前执行。

实施项：

- [x] 保持当前 HTTP API 合同；仅在本地功能复杂度需要时迁移 FastAPI。
- [x] 继续使用 SQLite；提供数据库备份、恢复与健康检查，不迁移 PostgreSQL。
- [x] 用本机线程和 SQLite 运行状态处理短任务；不引入 Redis 或远程队列。
- [x] 引入轻量工作流运行器；只有出现多步骤工具调用时才评估 LangGraph。
- [x] 按 MCP 设计本地只读工具注册，工具声明风险等级与启用状态；实际执行能力仍需逐工具授权。
- [x] 新增模型适配器接口，DeepSeek 已通过适配器接入，后续备用模型无需修改聊天主流程。
- [x] 验证：模型适配器、只读工具注册、轻量工作流与本地数据库备份已通过单元测试；未来新增工具时需补充该工具的超时与权限测试。

完成标准：新增一个本地模型或受控工具不需要修改聊天主流程，也不依赖外部基础设施。

## 5. Codex 执行方式

每个 P0-P4 阶段拆成一个独立任务，严格按以下循环执行：

1. 读取现有代码与数据模型，写明本次改动范围和验收标准。
2. 先补充或更新自动化验证，再实现最小改动。
3. 启动本地服务，使用真实 DeepSeek 和模拟失败场景各验证一次。
4. 检查浏览器中的连续对话、历史加载、刷新恢复与错误状态。
5. 将结果写入变更说明：改动文件、验证项、未解决风险。

建议的任务顺序：`P0-SSE 修复（已完成）` -> `P0-回归测试（已完成）` -> `P1-服务分层` -> `P1-运行详情` -> `P2-技能配置化` -> `P3-安全与备份`。P4 只在触发条件满足时执行。

## 7. 技术取舍

当前路线固定为“原生前端 + Python + SQLite + DeepSeek”的本地优先方案。FastAPI、LangGraph 与 MCP 只在本地实际需求出现后再引入；PostgreSQL、Redis 和云端任务队列不属于当前计划。

Codex 适合承担阶段性实现、回归验证、迁移与代码审查。平台内给终端用户提供的模型仍使用 DeepSeek，二者职责分离。

## 9. 单 Agent 核心演进计划

### 范围与原则

后续只建设**单 Agent**运行时，不引入多 Agent 协作、角色间通信或复杂任务市场。一个 Agent 负责理解任务、选择已授权技能与工具、执行计划、检查结果并输出答案。

模型不暴露私有思维链。前端只展示可审计的执行摘要，例如：使用的技能、计划步骤、工具调用状态、来源、错误与最终回答。

```text
对话输入（@技能 + 任务）
        |
        v
单 Agent Runtime
  ├─ 技能路由与权限校验
  ├─ 任务复杂度判断
  ├─ 计划执行 / 工具调用循环
  ├─ 结果检查
  └─ 运行记录与最终回答
        |
        +-- DeepSeek 模型
        +-- 本地工具与应用
        +-- 会话记忆与本地知识库
```

### P5：技能路由与执行上下文

目的：让技能从“提示词片段”演进为可控、可审计的单 Agent 输入能力。

实施项：

- [x] 支持消息内多选 `@技能`，并将选择写入用户消息和运行快照。
- [x] 用户关闭技能后，前端不展示、后端拒绝、运行时不注入，三层一致。
- [x] 定义 `Execution Context`：冻结本次模型、技能、应用、权限、超时与输入限制，后续步骤只读取该快照。
- [x] 增加技能路由规则：显式 `@技能` 永远优先；未显式选择时沿用会话默认技能，不自动越权调用。
- [x] 将技能区分为 `prompt_skill` 与 `tool_skill`：前者影响回答方式，后者可声明受控执行器和参数 Schema。
- [x] 在运行详情中增加“技能路由摘要”，展示本次技能、允许工具与执行计划。

完成标准：任一运行都可以确定回答使用了哪些技能、哪些应用配置和哪些权限；关闭或未授权的技能无法通过 UI、API 或模型文本绕过。

### P6：Plan-and-Solve 与可恢复执行

目的：让复杂单 Agent 任务先规划再执行，避免把所有工作压进一次模型回答。

实施项：

- [x] 定义复杂任务判定规则：多目标、需要工具、需要文件检索或用户明确要求计划时进入规划模式；普通问答直接回答。
- [x] 生成结构化计划：步骤 ID、目标和状态，并将其写入运行快照。
- [x] 使用 SQLite 持久化计划步骤状态；短任务失败可沿用现有消息重试。暂停/恢复仅在引入异步或写操作工具时实施。
- [x] 前端在运行详情展示计划和步骤状态，不展示私有推理过程。
- [x] 当前工具仅允许 `read_only` 风险等级；删除、覆盖文件和外部写操作尚未注册，未来注册前必须加入人工确认。

完成标准：一个多步骤任务可被中断并恢复；每一步状态、错误和最终产物均可追溯。

### P7：ReAct 工具调用与应用执行器

目的：让“应用”成为实际运行资源，而不仅是状态展示。

实施项：

- [x] 定义统一 Tool Contract：名称、说明、JSON 参数 Schema、返回 Schema、风险等级、超时与启用状态。
- [x] 为 DeepSeek 增加结构化工具调用适配；模型只能看到本次 `Execution Context` 允许的工具。
- [x] 实现单 Agent ReAct 循环：请求模型 -> 解析工具调用 -> 校验权限/参数 -> 执行 -> 记录观察结果 -> 继续或结束。
- [x] 首批只读本地工具：平台状态、工作区文件名检索；知识库检索在 P9 实现后接入。
- [x] 对工具调用设置最大步数、单工具超时和失败退出策略，避免无限循环。
- [x] 运行详情展示工具名、状态、计划和结果摘要，敏感输入不展示。

完成标准：模型可以在授权范围内可靠调用本地工具，工具失败不会导致运行长期卡住，也不会泄露敏感数据。

### P8：Reflection 与结果质量控制

目的：在输出前检查任务完成度，而不是展示或保存模型的思维链。

实施项：

- [x] 为需要高质量输出的任务定义检查项：目标覆盖、格式、冲突、未完成事项；普通问答不额外调用检查。
- [x] 单独生成结构化检查结果：`passed`、问题列表、摘要和修订次数；不保存模型私有推理文本。
- [x] 未通过时最多自动修订 1 次；检查或修订不可用时保留原回答，不让任务失败。
- [x] 前端显示可见的质量检查状态；运行详情保留检查摘要和修订次数。

完成标准：报告、方案和重要任务在交付前有可审计质量检查，且不存在无边界自动重试。

### P9：本地记忆与 RAG 知识库

目的：让单 Agent 可依据本地资料完成连续工作，同时保持个人本地部署边界。

实施项：

- [ ] 保留现有短期会话摘要；新增用户偏好与项目结论的长期记忆，并提供查看、编辑和删除入口。
- [x] 支持上传 Markdown、TXT、Word 资料，完成本地解析、切分、索引和删除；PDF 在本机安装 `pypdf` 后自动启用。
- [x] 建立本地检索接口；只有检索命中的片段才能作为知识依据进入模型上下文。
- [x] 回答中展示可点击资料名称和检索片段入口，不伪造来源。
- [x] 初期采用 SQLite 元数据加本地文件存储；向量索引方案在资料规模验证后再选择。

完成标准：用户可管理本地资料，Agent 能检索并引用资料回答，删除资料后不再被检索或注入。

### 推荐执行顺序

`P5 执行上下文与技能路由` -> `P7 首批只读工具` -> `P6 计划执行` -> `P8 质量检查` -> `P9 本地知识库`。

P6 与 P7 可根据实际需求调整顺序：优先需要文件/检索能力时先做 P7；优先需要复杂任务管理时先做 P6。多 Agent 协作不在当前路线中。

## 10. 模型效果优化执行计划

### 问题定义

当前平台的 Agent 机制已经具备执行上下文、工具和质量检查，但模型效果仍取决于模型路由、提示上下文、技能质量、资料依据和输出预算。后续优化不以增加更多 Agent 为目标，而以“同一任务达到接近或超过 DeepSeek 官方应用的完成度”为验收标准。

### P10：模型迁移与模型路由

目的：避免继续依赖即将退役的旧模型名称，并按任务复杂度选择合适的 DeepSeek 模型与模式。

## 13. Agent 平台能力深化路线（P28-P36）

目标：从“功能完整的单 Agent 平台”演进为“可评测、可记忆、可恢复、能稳定完成真实任务的个人工作 Agent”。本轮优先提升正确性与可验证性，不以增加模型、框架或页面数量作为完成标准。

开发方式：

- Codex 负责总体架构、计划维护、核心运行时与数据契约、主工作区集成、安全边界、代码审查、全量回归和最终验收。
